# Bold-Bose 项目全面优化计划

## 项目概述
NestJS + Next.js (Refine + Antd) Monorepo 管理后台系统，基于 Prisma ORM + PostgreSQL，模块涵盖系统管理、商城、会员、CMS、支付、微信公众号等。

---

## 2. 安全加固（高优先级）

### 2.1 CORS 配置
**问题**: `app.enableCors()` 无任何限制，生产环境暴露所有来源。
**修复**: 从环境变量读取允许的 origin 列表，生产环境限制为具体域名。

**文件**: `server/src/main.ts:13`

### 2.2 JWT 密钥强度
**问题**: JWT_SECRET 仅在运行时检查是否存在，无最小长度要求。
**修复**: 启动时校验 JWT_SECRET 长度 >= 32 字符。

**文件**: `server/src/modules/auth/jwt.strategy.ts:11-15`

### 2.3 密码强度校验
**问题**: 用户创建/修改密码时无强度校验。
**修复**: 在 User 相关 DTO 中添加密码复杂度校验（最少8位，包含大小写+数字）。

**文件**: 需新增 `server/src/modules/system/user/dto/` 中的密码校验

### 2.4 登录暴力破解防护
**问题**: 登录接口虽有全局 ThrottlerGuard，但无按用户名维度的限流。
**修复**: 在 auth controller 的 login 方法上增加按 IP + 用户名维度的限流装饰器，连续失败5次后锁定15分钟。

**文件**: `server/src/modules/auth/auth.controller.ts`

### 2.5 敏感信息日志泄露
**问题**: `pino-pretty` 在非生产环境输出 debug 级别日志，可能包含敏感数据。
**修复**: 确保密码、token、API secret 等字段在日志中被脱敏。

**文件**: `server/src/app.module.ts:28-30`, `server/src/main.ts`

### 2.6 Session 表缺少唯一约束
**问题**: `UserSession` 表的 `token` 字段虽有 `@unique`，但缺少 `userId` 索引，清理效率低。
**修复**: 添加 `@@index([userId])` 和 `@@index([expiresAt])`。

**文件**: `server/prisma/schema.prisma:315-330`

## 3. 后端性能优化（中优先级）

### 3.1 JWT 验证的 N+1 查询
**问题**: `JwtStrategy.validate()` 每次请求执行 2 次数据库查询（session + user），且 lastActiveTime 更新虽有 60s 节流，但仍是 fire-and-forget 异步更新。
**优化**:
- 将 user 信息缓存到 `UserCacheService` 中，复用已有的缓存基础设施
- session 查询结果也应缓存（TTL 短，如 10s），减少高频请求的 DB 压力

**文件**: `server/src/modules/auth/jwt.strategy.ts`

### 3.2 前端 `getPermissions` / `getIdentity` 重复请求
**问题**: `auth-provider.ts` 中 `getPermissions` 和 `getIdentity` 各自独立调用 `/system/auth/get-permission-info`，Refine 框架可能在初始化时分别调用，导致 2 次相同请求。
**优化**: 合并为一次请求，或使用 Refine 的 `getPermissions` 同时返回 user 和 permissions。

**文件**: `admin/src/providers/auth-provider.ts:57-70`

### 3.3 操作日志写入阻塞
**问题**: `LogInterceptor.saveLog()` 在请求完成后同步写入数据库，如果 DB 慢会延迟响应。
**优化**: 使用 `@nestjs/schedule` 或消息队列异步批量写入操作日志。

**文件**: `server/src/shared/interceptors/log.interceptor.ts:51-83`

### 3.4 SPU 更新时的全量删除重建
**问题**: `SpuService.update()` 先 `deleteMany` 再 `create` 所有 SKU，效率低且可能丢失未变更的数据。
**优化**: 对比新旧 SKU 列表，仅更新变更的、创建新增的、删除移除的。

**文件**: `server/src/modules/mall/spu/spu.service.ts:91-142`

### 3.5 缓存策略不足
**问题**: 仅 `UserCacheService` 使用了 cache-manager，商品、分类、品牌等高频读取数据未缓存。
**优化**: 为 MallSpu（热门商品）、MallCategory（分类树）、MallBrand（品牌列表）添加缓存层。

**新增文件**: `server/src/modules/mall/shared/mall-cache.service.ts`

## 4. 前端优化（中优先级）

### 4.1 Next.js 配置优化
**问题**: `next.config.ts` 未配置 `output: 'standalone'`（部署优化）、未启用图片优化等。
**优化**: 添加 standalone output、图片域名白名单、压缩等配置。

**文件**: `admin/next.config.ts`

### 4.2 缺少 `loading.tsx` / `error.tsx`
**问题**: 无 Next.js 的 loading 和 error 边界文件，用户在页面加载时无良好反馈。
**优化**: 为 `(dashboard)` 路由组添加 `loading.tsx` 和 `error.tsx`。

**新增文件**: `admin/src/app/(dashboard)/loading.tsx`, `admin/src/app/(dashboard)/error.tsx`

### 4.3 Axios 无请求取消
**问题**: 切换页面时未取消正在进行的请求，可能导致内存泄漏或数据竞争。
**优化**: 使用 AbortController 或 axios CancelToken 在组件卸载时取消请求。

**文件**: `admin/src/lib/axios.ts`

## 5. DevOps 与部署优化（中优先级）

### 5.1 Docker Compose 安全性
**问题**: PostgreSQL 密码硬编码为 `postgres`，无健康检查，无资源限制。
**修复**:
- 使用 `.env` 文件注入密码
- 添加 `healthcheck`
- 添加 `deploy.resources.limits` 内存/CPU 限制

**文件**: `docker-compose.yml`

### 5.2 缺少生产环境 Dockerfile
**问题**: 无 server 和 admin 的 Dockerfile，部署依赖手动操作。
**优化**: 创建多阶段构建 Dockerfile。

**新增文件**: `server/Dockerfile`, `admin/Dockerfile`

### 5.3 环境变量管理
**问题**: `.env` 文件未被 `.gitignore` 排除（仅排除了 `yudao-cloud-master`）。
**修复**: 确保 `.env`、`.env.local` 在 `.gitignore` 中。

**文件**: `.gitignore`

### 5.4 缺少健康检查端点
**问题**: `AppController` 无 `/health` 端点，Docker/K8s 无法做存活探针。
**优化**: 添加 `/health` 端点返回 DB 连接状态。

**文件**: `server/src/app.controller.ts`

## 6. 代码质量与可维护性（低优先级）

### 6.1 缺少 ESLint/Prettier 统一配置
**问题**: root `package.json` 无 lint/format 脚本，server 和 admin 各自独立配置。
**优化**: 在 root 添加统一的 `lint`、`format` 脚本。

**文件**: `package.json`

### 6.2 测试覆盖率不足
**问题**: 仅 `auth.service.spec.ts`、`user-cache.service.spec.ts`、`job.service.spec.ts` 有测试，大量 service 无单元测试。
**优化**: 优先为订单、支付、SPU 等核心业务逻辑添加测试。

### 6.3 `yudao-cloud-master` 目录
**问题**: Java 版本参考项目在仓库中但已被 `.gitignore` 排除，建议完全移除以减小仓库体积。
**优化**: 从磁盘删除 `yudao-cloud-master/` 目录。

## 实施优先级

| 阶段 | 内容 | 影响 |
|------|------|------|
| P0 | 数据库索引 + CORS + JWT密钥校验 | 安全 + 性能基础 |
| P1 | Session清理索引 + 登录限流 + 前端重复请求 | 安全 + 性能 |
| P2 | 操作日志异步化 + SPU SKU优化 + 商品缓存 | 性能 |
| P3 | Docker加固 + Dockerfile + 健康检查 | 部署 |
| P4 | 前端优化 + 代码质量 + 测试 | 可维护性 |

## 验证方式

1. **数据库索引**: 运行 `npx prisma migrate dev` 生成迁移，用 `EXPLAIN ANALYZE` 验证查询计划
2. **安全**: 使用 curl 测试 CORS 头、用短密码测试注册是否被拒绝
3. **性能**: 使用 `ab` 或 `k6` 对 `/admin-api/mall/spu` 做压测，对比优化前后 QPS
4. **Docker**: `docker compose config` 验证配置，`docker compose up` 验证健康检查
5. **前端**: `npm run build` 验证构建无错误，检查 bundle size
