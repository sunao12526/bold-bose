# Bold-Bose 项目代码评审报告

## 项目概述

NestJS + Next.js (Refine + Ant Design 5) 电商后台管理系统 Monorepo，包含系统管理、基础设施、商城、会员等模块。PostgreSQL + SeaweedFS 作为存储层，Prisma 作为 ORM。

---

## 一、安全性问题 (Critical)

### 1. JWT Secret 硬编码 [HIGH]
- **文件**: `server/src/modules/auth/jwt.strategy.ts:12`
- **问题**: JWT secret 硬编码为 `'yudao-nestjs-secret-key-2026'`，`process.env.JWT_SECRET` 仅作 fallback，但 `.env` 中并未设置该变量
- **修复**: `.env` 中添加 `JWT_SECRET=<强随机字符串>`，移除硬编码 fallback 或仅在开发环境使用

### 2. 文件上传接口无鉴权 [HIGH]
- **文件**: `server/src/modules/infra/file/file.controller.ts:14-16`
- **问题**: `uploadFile` 被标记为 `@Public()`，任何人都可以无认证上传文件，可能导致恶意文件上传或磁盘耗尽
- **修复**: 移除 `@Public()` 装饰器，或添加文件大小限制和文件类型白名单

### 3. 数据库凭据硬编码 [MEDIUM]
- **文件**: `server/.env:12`, `docker-compose.yml:8-9`
- **问题**: PostgreSQL 用户名密码为 `postgres/postgres`，且 `.env` 已被 git 跟踪
- **修复**: 确保 `.gitignore` 包含 `.env`，生产环境使用强密码

### 4. CORS 全开 [MEDIUM]
- **文件**: `server/src/main.ts:11`
- **问题**: `app.enableCors()` 无配置，允许任何域跨域请求
- **修复**: 生产环境应配置 `origin` 白名单

### 5. 前端登录页硬编码默认密码 [LOW]
- **文件**: `admin/src/app/login/page.tsx:10`
- **问题**: `initialValues: { username: 'admin', password: 'admin123' }` 在源码中暴露了默认密码
- **修复**: 移除初始值或在生产环境禁用

### 6. 前端 API 地址硬编码 [LOW]
- **文件**: `admin/src/lib/axios.ts:4`, `admin/src/providers/data-provider.ts:44`
- **问题**: `baseURL` 和 `getApiUrl` 硬编码为 `http://localhost:3000/admin-api`
- **修复**: 使用环境变量 `process.env.NEXT_PUBLIC_API_URL`

## 二、业务逻辑缺陷

### 7. 订单取消不回滚库存 [HIGH]
- **文件**: `server/src/modules/mall/order/order.service.ts:87-99`
- **问题**: `cancel()` 只更新订单状态为 CANCELLED，但不回滚 SPU 的 `totalStock` 和 SKU 的 `stock`
- **修复**: 取消订单时恢复对应 SKU 的库存

### 8. 退款审批不回滚库存 [HIGH]
- **文件**: `server/src/modules/mall/refund/refund.service.ts:39-76`
- **问题**: `approve()` 将余额退还给会员，但不回滚 SPU/SKU 库存
- **修复**: 审批退款时恢复对应 SKU 的库存

### 9. 会员调整数据非原子操作 [MEDIUM]
- **文件**: `server/src/modules/member/member.service.ts:38-49, 51-62, 64-78`
- **问题**: `adjustPoints`/`adjustBalance`/`adjustExperience` 先读取再计算再写入，存在竞态条件，可能被并发请求覆盖
- **修复**: 使用 Prisma 的 `increment`/`decrement` 操作，或使用数据库事务 + 乐观锁

### 10. 角色/用户删除无关联检查 [MEDIUM]
- **文件**: `server/src/modules/system/role.service.ts:32-33`, `server/src/modules/system/user.service.ts:78-83`
- **问题**: 删除角色时不检查是否有用户关联（仅有 `super_admin` 禁删保护），删除用户不检查是否有未完成订单
- **修复**: 删除前检查关联数据

### 11. 菜单删除子菜单检查不完整 [LOW]
- **文件**: `server/src/modules/system/menu.service.ts:29-35`
- **问题**: 使用 `findFirst` 只检查第一层子菜单，但 Prisma schema 中 `onDelete: Cascade` 已经会级联删除子菜单，所以这个检查实际在阻止所有有子菜单的菜单被删除，但检查逻辑与实际行为不一致
- **修复**: 移除手动检查（依赖 Cascade），或改为检查是否有关联的按钮权限

## 三、性能问题

### 12. PermissionsGuard 每次请求多次数据库查询 [HIGH]
- **文件**: `server/src/shared/guards/permissions.guard.ts:13-57`
- **问题**: 每个受保护的请求会执行 2 次数据库查询（`userRole.findMany` + `roleMenu.findMany`），且 `super_admin` 也需要先查数据库
- **修复**: 
  - 在 JWT payload 中嵌入 `roles` 字段，避免每次查数据库
  - 或在 JWT Strategy 中将 roles 缓存到 request 上

### 13. JWT Strategy 每次请求多次数据库查询 [HIGH]
- **文件**: `server/src/modules/auth/jwt.strategy.ts:17-57`
- **问题**: 每次请求执行 2 次数据库查询（`userSession.findUnique` + `user.findUnique`）
- **修复**: 使用 Redis 缓存 session，或在 JWT 中包含足够信息减少查询

### 14. 接口无分页支持 [MEDIUM]
- **文件**: 多个 service 的 `findAll` 方法
- **问题**: `findAll` 全部返回所有数据（`user.service.ts:34-49`, `role.service.ts:12-15`, `order.service.ts:9-18` 等），无分页、排序、搜索参数
- **修复**: 添加分页参数（`page`/`pageSize`），支持排序和条件筛选

### 15. 代码生成器 SQL 注入风险 [MEDIUM]
- **文件**: `server/src/modules/infra/codegen/codegen.service.ts:13-24, 39-46`
- **问题**: `$queryRawUnsafe` 直接拼接 SQL，虽然当前是内部调用，但 `getDbTables()` 的查询没有参数化表名过滤
- **修复**: 对动态部分使用参数化查询

## 四、代码质量问题

### 16. 大量使用 `any` 类型
- **涉及文件**: 几乎所有 controller 和 service
- **问题**: `@Body() data: any`, `create(data: any)`, `update(id: number, data: any)` 等，缺少 DTO 类定义和验证
- **修复**: 为每个接口定义 DTO 类并使用 `class-validator` 验证（已有 LoginDto 的良好实践，应推广）

### 17. SPU 创建/更新逻辑重复
- **文件**: `server/src/modules/mall/spu/spu.service.ts`
- **问题**: `create()` 和 `update()` 中 SKU 聚合计算逻辑完全重复（第 12-25 行 vs 第 76-89 行）
- **修复**: 提取为私有方法 `calculateSkuAggregates(skus)`

### 18. FileService.remove 重复创建客户端实例
- **文件**: `server/src/modules/infra/file/file.service.ts:56-74`
- **问题**: `remove()` 方法中手动创建 `LocalFileClient`/`S3FileClient`，而 `FileConfigService` 已有 `getMasterClient()`
- **修复**: 复用 `FileConfigService`，或注入文件客户端工厂

### 19. 前端 header 组件轮询无清理保障
- **文件**: `admin/src/components/header.tsx:32-37`
- **问题**: `setInterval` 30 秒轮询通知，但如果组件在 SSR 阶段渲染，`useEffect` 可能不会正确清理
- **修复**: 当前实现已使用 cleanup function，但可添加 `isMounted` 检查

### 20. 签到连续天数逻辑硬编码为 7
- **文件**: `server/src/modules/member/sign-in.service.ts:93-95`
- **问题**: 连续签到最大天数硬编码为 7，应从配置中读取
- **修复**: 动态查询 `memberSignInConfig` 的最大天数

## 五、架构改进建议

### 21. 缺少测试文件
- `server/test/` 目录下仅有 `jest-e2e.json`，无实际测试用例
- 建议至少为 Auth、User、Order 等核心模块编写单元测试和 E2E 测试

### 22. 缺少 Rate Limiting
- 登录接口无频率限制，易受暴力破解攻击
- 建议使用 `@nestjs/throttler` 添加全局或局部速率限制

### 23. 缺少输入长度验证
- DTO 中仅有 `@IsNotEmpty`，缺少 `@MaxLength`/`@MinLength` 限制
- 如 `username` 在 Prisma schema 中定义为 `VarChar(50)`，但 DTO 未限制长度

### 24. 密码策略过弱
- `LoginDto` 和用户创建接口无密码强度验证
- 建议添加最小长度、复杂度要求
