# 项目优化建议

## 项目概述
NestJS + Next.js (Refine + Antd) 电商管理系统 Monorepo，包含 System/Mall/Member/Pay/CMS/MP/Infra 七大模块。

---

## 一、安全问题 (高优先级)

### 1.1 移除硬编码的 JWT_SECRET
- **文件**: `server/.env`
- **问题**: JWT_SECRET 已提交到 Git 仓库
- **建议**: 使用环境变量注入，不将密钥提交到版本控制；添加 `.env` 到 `.gitignore`

### 1.2 移除硬编码的 URL
- **文件**: `server/src/modules/mall/order/order.service.ts:72`
- **问题**: `http://localhost:3000` 写死在代码中
- **建议**: 使用配置模块（`@nestjs/config`）管理环境相关的 URL

### 1.3 添加请求限流
- **建议**: 引入 `@nestjs/throttler` 防止暴力攻击，特别是登录、短信验证码等接口

### 1.4 补充 .gitignore
- **当前**: 仅忽略 `/yudao-cloud-master`
- **建议**: 添加 `node_modules/`, `dist/`, `.env`, `coverage/` 等标准忽略项

---

## 二、代码质量 (高优先级)

### 2.1 消除 UserAgent 解析重复
- **文件**: `server/src/modules/auth/auth.service.ts`
- **问题**: `parseUserAgent` 方法在 `login()` 和 `socialLogin()` 中重复实现
- **建议**: 提取为共享工具函数

### 2.2 移除前端 console.log
- **文件**: `admin/src/providers/data-provider.ts:6`
- **问题**: `console.log('FRONTEND GETLIST:', ...)` 留在生产代码中
- **建议**: 使用条件日志或移除

### 2.3 添加 DTO 类型验证
- **文件**: `server/src/modules/mall/spu/spu.service.ts:8,74`
- **问题**: 大量使用 `any` 类型，缺乏输入验证
- **建议**: 为所有 Controller 添加 class-validator DTO

### 2.4 统一日志策略
- **当前**: 混用 `console.error` / `console.warn` / `console.log`
- **建议**: 统一使用 NestJS Logger，或引入 pino/winston 结构化日志

---

## 三、测试覆盖 (高优先级)

### 3.1 当前状态
- 仅有 1 个测试文件 (`app.controller.spec.ts`)
- 无 Service 单元测试
- 无 E2E 测试

### 3.2 建议优先编写测试的模块
1. **支付模块** (`pay/pay-order.service.ts`): 资金相关，必须有测试保障
2. **订单模块** (`mall/order/order.service.ts`): 涉及库存回滚事务
3. **认证模块** (`auth/auth.service.ts`): 安全关键路径
4. **签到模块** (`member/sign-in.service.ts`): 涉及积分计算

---

## 四、性能优化 (中优先级)

### 4.1 缓存层升级
- **文件**: `server/src/shared/user-cache.service.ts`
- **问题**: 内存 Map 缓存，重启丢失，无分布式支持
- **建议**: 考虑 Redis 缓存（可选），或至少添加 LRU 淘汰策略

### 4.2 添加分页
- **文件**: `server/src/modules/mall/order/order.service.ts:17` (`findAll`)
- **文件**: `server/src/modules/mall/spu/spu.service.ts:54` (`findAll`)
- **问题**: 无分页，大数据量时性能差
- **建议**: 实现统一的分页 DTO 和分页查询

### 4.3 数据库索引优化
- **问题**: 仅依赖 Prisma 默认索引
- **建议**: 为高频查询字段添加复合索引（如 `mall_orders(member_id, status)`, `mall_skus(spu_id)`）

### 4.4 异步日志写入
- **文件**: `server/src/shared/interceptors/log.interceptor.ts`
- **问题**: 日志同步写入数据库，影响请求响应时间
- **建议**: 使用队列（Bull/BullMQ）异步写入日志

---

## 五、架构优化 (中优先级)

### 5.1 引入 API 版本控制
- **建议**: 在 `main.ts` 中配置 `app.enableVersioning()`，使用 URI 版本 `/v1/...`

### 5.2 统一响应格式
- **建议**: 创建全局响应拦截器，统一返回 `{ code, message, data }` 结构

### 5.3 配置外部化
- **当前**: 硬编码的 24h token 过期时间、mock URL 等
- **建议**: 所有可配置项提取到 `.env` 或配置模块

### 5.4 前端 API URL 可配置
- **文件**: `admin/src/providers/data-provider.ts:49`
- **问题**: `getApiUrl` 硬编码 `http://localhost:3000`
- **建议**: 使用 `NEXT_PUBLIC_API_URL` 环境变量

---

## 六、DevOps (低优先级)

### 6.1 生产环境配置
- **建议**: 添加 `docker-compose.prod.yml`，配置生产级 PostgreSQL、环境变量、健康检查

### 6.2 CI/CD
- **建议**: 添加 GitHub Actions 配置，自动化测试、lint、build

### 6.3 Docker 优化
- **建议**: 创建多阶段 Dockerfile，减小生产镜像体积
